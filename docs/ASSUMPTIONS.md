# Assumptions and Open Decisions

This register allows implementation to proceed without inventing external facts. Validate assumptions before production; an assumption is not a commitment from TERRAST, JPX, SSBJ, ISSB, Supabase, OpenAI, or any customer.

| ID    | Working assumption                                                                                                                        | Reason / impact                                                     | Validation needed                                                                   |
| ----- | ----------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------- | ----------------------------------------------------------------------------------- |
| A-01  | The deliverable is a concept MVP, not a JPX-approved or JPX-provided service.                                                             | Required legal/positioning boundary; disclaimer appears throughout. | Written partnership/communications approval before any wording change.              |
| A-02  | No usable TERRAST production API contract has been supplied.                                                                              | API connector remains fail-closed; mock/import only.                | Authorized interface-control document and sandbox access.                           |
| A-03  | Synthetic demo records may resemble generic sustainability fields but never real TERRAST/customer records.                                | Enables offline demo without data rights risk.                      | Dataset owner and privacy review before external demo.                              |
| A-04  | `organizations` is the tenant root; immutable `tenant_id` is the boundary and child rows use `organization_id`.                           | Supports multi-company evolution while keeping RLS consistent.      | Product/security review of group-company tenancy and data residency.                |
| A-05  | One user can have multiple organization roles; active membership is authoritative.                                                        | Supports reviewer/assurer/operator scenarios.                       | Confirm segregation-of-duties and last-admin rules.                                 |
| A-05a | One company organization owns one company in the MVP.                                                                                     | Organization-wide RLS is therefore company-scoped.                  | Add member-to-company scope before enabling group tenants.                          |

## Unresolved decisions that block production

- TERRAST interface/auth/data-rights/SLA and operational ownership.
- Formal relationship and approved claims/branding for JPX or any exchange.
- Production tenant/group model, region, retention, deletion, legal hold, backup/PITR, RTO/RPO.
- SSO/MFA/SCIM identity lifecycle and segregation of duties.
- Data classification, privacy notice/consent, DPA/subprocessors, OpenAI use, and cross-border transfer.
- Authoritative SSBJ/ISSB catalog curation and emission-factor licensing.
- Aggregate cohort/privacy thresholds and company consent wording.
- Malware scanning, WAF/distributed rate limiting, SIEM/incident response, external penetration test.
- Production domain, Supabase project, administrators, billing, and on-call ownership. The GitHub repository and Vercel demo project are already created.
